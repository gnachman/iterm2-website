2026-06-26: ChatViewController: fix oversized gaps when the

2026-06-23: Companion: cache keychain identity, primed at
launch

The Noise keypair, paired phone key, and room secret were
read from the login keychain on the connection/listening hot
path. The login keychain gates each item by the signature of
the binary that wrote it, so a build signed differently from
the writer pops a confirmation prompt. That prompt could
appear mid-connection, with nobody at the keyboard,
defeating the point of the phone app.

read the cache and never touch the keychain.
primeCacheAtLaunch is side-effect-free: it does not generate
a Noise keypair, so an unpaired user gets no new key
material.

The cache-miss writeback only fills a still-absent entry, so
a concurrent store/delete cannot be clobbered by a stale
keychain read.

2026-06-22: Show the tty-attached foreground job as the job
name

The job name in titles, the session icon, and automatic
profile switching are derived from the deepest foreground
job. When a program like claude runs helper subprocesses
(MCP servers, caffeinate) those run in the foreground
process group but talk to their parent over pipes, so the
deepest foreground job is the helper, not the interactive
program. That made tabs show "(chrome-devtools-mcp)" or
"(caffeinate)" instead of the real job.

Pick the deepest foreground job whose stdin or stdout is
actually the session tty. The process cache derives the
session tty once per session from the root pid's stdio
(searching past a root-owned login whose fds we cannot
read), keyed by pid and start time so a recycled pid
re-derives. Device lookups are restricted to /dev/tty* so a
redirection to /dev/null is not mistaken for the terminal.
The raw deepest job remains available as the new deepestJob
variable, and trigger/orchestration ancestry still uses the
full chain.

Remote (SSH integration) sessions lack fd data and fall back
to the old behavior. Issue 12898.

2026-06-19: Preserve folds when a non-alt-screen program
repaints from the top

Programs like Claude Code that do not use the alternate
screen buffer repaint on launch by homing the cursor and
drawing over whatever was on screen. The existing
saveScreenToScrollbackIfCursorMovedAboveCommandOutput
preserves content in that case, but only with shell
integration, and it bails when the program drew before
homing (appendedTextSinceCommandExecuted is set), which is
exactly what Claude does. When a fold sat in the overwritten

leaving an orphaned unfold icon in the margin.

works without shell integration and regardless of whether
the program has already drawn. When the cursor moves at or

content above it into history, preserving the fold mark at
its absolute position.

events that change the fold set or a fold's absolute
coordinate, caught centrally at two chokepoints
(shiftIntervalTreeObjectsInRange: for relocations and
didRemoveObjectFromIntervalTree: for removals) plus the
rebuild and swap paths that bypass them (reflow,
clearScrollbackBuffer, removeLastLine, deserialization,
alt-screen and tmux grid swaps), and recomputed lazily.

2026-06-09: Fix dropped completion handler in
iTermAsyncStringConversion

The conversion is dispatched to a background queue from
init, but callers set completionHandler after init returns
(both VT100Parser and the unit test do). If the background
work finished before the handler was assigned, the handler
was never called: VT100Parser's sOutstandingPreconvertBytes
backpressure counter would never be decremented for that
conversion, and testAsyncCompletionHandlerCalled would time
out. The race was timing-dependent and only showed up on the
slower CI scheduler.

whichever side loses the race: the background completion
fires the handler if it is already set, and the setter fires
it if the work already finished. Also makes
completionHandler access thread-safe (previously read on the
background queue while written on the caller thread).

2026-06-01: AI: Anthropic prompt caching on system + tools

Two breakpoints, not one, by intent. Anthropic's prefix

  - Marker on the last tool covers [tools] only. Fallback
  segment
    that still hits when the system prompt changes (a future
    per-turn system addendum, for example).
  - Marker on system covers [tools + system]. Common-case
  longest
    match: both stable for the chat, the whole prefix

2026-05-28: AI chat: repair orphan tool_result on chat
reload

An auto-approved ("always") remote command historically
squelched the remoteCommandRequest from the persisted
transcript while keeping the remoteCommandResponse. On
reload, ChatAgent.load(messages:) rebuilt the prompt from
that transcript and the result was an orphan tool_result
with no matching tool_use, which Anthropic rejects with 400
'unexpected tool_use_id found in tool_result blocks'. The
bug predates v3.6.10, so shipping conversations on disk
already contain orphans.

AIChatToolCallRepair walks the rebuilt prompt and, when it
finds an orphaned functionOutput, inserts a synthesized
assistant functionCall immediately before it. Two pairing
modes have to be satisfied because they reflect what the
vendor serializers actually do:

  * Id-based (Anthropic, OpenAI Responses): match on call
  id. The inner
    FunctionCall.id MUST equal the wrapper
    FunctionCallID.callID; CompletionsAnthropic keys
    tool_use blocks off the inner id and ignores the
    wrapper, so a nil inner id would serialize the synth
    call as plain text and reproduce the 400.

  * Position-based (Gemini, legacy OpenAI function_call):
  the
    deserializer sets both the inner FunctionCall.id and the
    wrapper FunctionCallID to nil (Gemini.swift parser).
    Those nils flow into the persisted
    .remoteCommandResponse and arrive here as a
    functionOutput with no id at all. Such a result cannot
    be paired by id, so synthesize a nil-id functionCall and
    rely on adjacency. The previous nil-id drop was silently
    dropping every Gemini tool result on reload, leaving an
    orphan functionCall instead of an orphan
    functionResponse.

Also fixes a second-order bug in role(from:) that the new
live coverage caught: tool results were being given
LLM.Role.user (because the chat Message author is .user),
but DeepSeek's serializer keeps the wire role as 'user' (and
rejects the request as 'insufficient tool messages') while
OpenAI Responses logs 'Unexpected user message body' and
silently drops the item. Production live calls already
construct tool results with role=.function (AITerm.swift
around line 957); this brings the reload path in line.
Anthropic and Gemini accept either.

2026-05-19: API server: advisory-lock socket dir to prevent
second iTerm2 from clobbering

Two iTerm2 processes sharing an application-support
directory (same -suite, or both with no -suite) would race
in listenOnUnixSocket: the second to start unconditionally
unlink()s the path and bind()s its own socket, leaving the
first listening on an orphaned inode that no client can
reach. Take a non-blocking flock on private/socket.lock
before unlink+bind so the second instance refuses to start
the API server instead.

2026-05-07: Fix tmux send-keys regression: route C0 control
bytes through -H

2026-05-01: Fix shell-arg truncation and POSIX/bash quoting
bugs

componentsInShellCommand was running every unquoted argv
element through -stringByExpandingTildeInPath, which
silently caps at PATH_MAX (1024). Long Code Review launches
died with zsh "unmatched '" because the trailing

chars and apostrophes survive the round-trip.

Normalize CR/CRLF to LF in the Code Review prompt wrapper so

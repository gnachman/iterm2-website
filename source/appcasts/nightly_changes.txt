2026-03-31: Attempt to fix tmux resize on attach for issue
9480

Each time you attach to a tmux -CC session, the window goes
through multiple sequential resize operations. The tmux
layout specifies a pixel size for the tab content area, but
this size is derived by rounding down to whole cells  so
it's almost always slightly smaller than the current
window's content area.  Each resize shaves off that
fractional-cell remainder, and the loss accumulates across
attach/detach cycles.

The original fix (lazyFitWindowToTabSize) correctly
suppressed this sub-cell shrinkage, but it only protected
one of the three resize paths that fire during attach:

1. openTabWithTmuxLayout calls lazyFitWindowToTabSize
decides not to shrink (protected) 2. appendTab triggers
tabViewDidChangeNumberOfTabViewItems which calls
fitWindowToTabs  shrinks the window (unprotected) 3.
loadTmuxLayout calls fitWindowToTabs  shrinks the window
again (unprotected)

Path 2 undoes path 1's decision before loadTmuxLayout even
gets to path 3. So no matter what path 1 decided, the window
still shrinks.

The fix applies the same lazy fitting logic to all three
paths, so sub-cell shrinkage is blocked regardless of which
code path triggers the resize.

2026-03-05: Fix find next (Cmd+G) skipping matches after

When tail find searched from a mid-line position (e.g., x=50
on line 12), it would report having searched the coordinate
range (50,12) to (2,37).  The code then removed ALL search
results on lines 12-37, but tail find only re-discovered

drop and Cmd+G to skip those matches.

converted VT100GridAbsCoordRange to NSRange by discarding
the x-coordinate: NSMakeRange(range.start.y, range.end.y -
range.start.y)

2026-07-13: Retain foreground-job ancestors via strong
parent links

iTermProcessInfo.parent was weak, but callers hold a
deepest-job iTermProcessInfo past the lifetime of the

_lastProcessInfo, and the process cache swaps in a fresh
collection on its work queue. The ancestor infos are owned

parent chain dangled to nil and foregroundJobAncestorNames

through the weak collection, so there is no parent -> child
strong edge.

Adds ForegroundJobAncestryTests reproducing the collapse
(deepest job read after its collection is freed) and a
contrast test showing a dropped node in a live collection
over-collapses instead, ruling out the naive theory.

2026-07-11: setsid the launched iTermServer so a signal to
iTerm2 can't kill it

iTermServer", which keeps it a child of iTerm2 for issue
9276 so tools can walk the shell's parent chain to the app).
That commit removed the double-fork's setsid without a
replacement, and the server-launch path (fork + iTermExec)
never had its own setsid, so the server inherited iTerm2's
session and process group.

When iTerm2 is launched from a terminal, it sits in that

fork fresh shells instead.

gets its own session and process group. This keeps the
parent pointer at iTerm2 (preserving the 9276 ancestry
behavior) while making the server immune to terminal
job-control signals.

2026-07-11: Session Status merge: represent a workgroup by
its visible peer

With "Merge workgroup statuses" on, the merged row picked
its representative by recency (the most recently changed
peer wins), falling back to priority only on a timestamp
tie. In a chat plus code-review workgroup that surfaces the
wrong peer: a busy code-review peer holding a steady

and above recency so a stable-status busy peer keeps the
row. When there is no visible switcher peer (every member
buried, or a non-switcher group) the prior recency/priority
rules still apply.

For the record: the original merge commit (3c31c6f82)
message claimed the representative was the highest-priority
status with ties broken by the most recent update, but the
shipped code was recency-first from the start. The two never
matched; this makes the intended behavior explicit.

2026-07-02: Run SSH integration through the user's login
shell

The SSH command type launched it2ssh via a bare
`/usr/bin/login -fpq <user> <shell> -c ...`, which is a
non-login, non-interactive shell and so sourced no dotfiles.
Any environment the user configures in their shell startup
(most notably a custom SSH_AUTH_SOCK pointing at an agent
that holds their keys) was therefore not applied, and ssh
fell back to the default agent. For agent-backed keys
(Secretive, 1Password, a manually run ssh-agent, etc.) this
produced "Load key ...pub: invalid format" followed by
"Permission denied (publickey)".

shell" path does, using ShellLauncher so the shell is exec'd
with argv[0] = "-<basename>" (login) and -i (interactive),
sourcing both profile and rc files uniformly across bash,
zsh, fish, tcsh, and xonsh. it2ssh and the ssh binary it
invokes are referenced by absolute path, so a dotfile that
rewrites $PATH cannot stop them from launching. Dotfile
output printed before it2ssh runs is handled as ordinary
terminal output; the conductor parser does not engage until
the token-gated it2ssh= OSC arrives, so it cannot desync the
protocol.

Fixes the SSH-command-type case of issue #12909.

2026-06-29: Add RLog retrospective logging and migrate
low-frequency DLog sites

RLog behaves like DLog when debug logging is on, but when it
is off it retains its message in a bounded 10 MB in-memory
ring buffer instead of dropping it, so the lead-up to a
low-frequency event can be recovered even when nobody
enabled debug logging ahead of time.

SwiftDebugLogging.swift): - RLog macro (ObjC) and RLog
function (Swift). Unlike DLog, RLog
  evaluates its message unconditionally, so it is only for

  version takes a plain String rather than an @autoclosure
  to make that eager evaluation explicit.
- iTermRetrospectiveLogString / iTermClearRetrospectiveLog
expose and
  clear the ring. The ring is never written into debug logs
  automatically (the user did not opt into capturing it);
  retrieval is an explicit action via a new iTerm2 > Save
  Retrospective Debug Logs menu item that writes it to a
  file the user chooses.

Migration: audited all ~8.6k DLog statements and converted
the low-frequency, diagnostically valuable ones with cheap
arguments to RLog (Tier A + Tier B from the audit), ~1500
sites.

Left as DLog where a blind swap would be wrong: - Code
running in a separate process, where the main-app ring is
never
  surfaced and RetrospectiveLogImpl is not linked

  worker).
- Conductor.swift, where DLog is a verbose-gated instance

  adds an address/depth prefix rather than the global

  descriptions, dumps), which must not run unconditionally.

2026-06-23: Companion: cache keychain identity, primed at
launch

The Noise keypair, paired phone key, and room secret were
read from the login keychain on the connection/listening hot
path. The login keychain gates each item by the signature of
the binary that wrote it, so a build signed differently from
the writer pops a confirmation prompt. That prompt could
appear mid-connection, with nobody at the keyboard,
defeating the point of the phone app.

primed once at launch (while the user is present, so any
prompt is answered then).  The connection and push paths now
read the cache and never touch the keychain.
primeCacheAtLaunch is side-effect-free: it does not generate
a Noise keypair, so an unpaired user gets no new key
material.

The cache-miss writeback only fills a still-absent entry, so
a concurrent store/delete cannot be clobbered by a stale
keychain read.

2026-06-22: Show the tty-attached foreground job as the job
name

The job name in titles, the session icon, and automatic
profile switching are derived from the deepest foreground
job. When a program like claude runs helper subprocesses
(MCP servers, caffeinate) those run in the foreground
process group but talk to their parent over pipes, so the
deepest foreground job is the helper, not the interactive
program. That made tabs show "(chrome-devtools-mcp)" or
"(caffeinate)" instead of the real job.

Pick the deepest foreground job whose stdin or stdout is
actually the session tty. The process cache derives the
session tty once per session from the root pid's stdio
(searching past a root-owned login whose fds we cannot
read), keyed by pid and start time so a recycled pid
re-derives. Device lookups are restricted to /dev/tty* so a
redirection to /dev/null is not mistaken for the terminal.
The raw deepest job remains available as the new deepestJob
variable, and trigger/orchestration ancestry still uses the
full chain.

Remote (SSH integration) sessions lack fd data and fall back
to the old behavior. Issue 12898.

2026-06-22: Orchestrator: screen-check backstop for stuck
tab-status watchers

A tab-status watcher waits indefinitely on a status
transition. If the cc-status hook drops the event that would
report the target state, that transition never arrives and
the watch hangs (a session reported "working" that has
actually gone idle), as seen when a watcher waited all night
without firing.

Add a screen-observation backstop. When a tab-status watcher
is registered (register_watch / start_code_review), arm a
timer that, once the session's rendered screen has been
quiet for 5 minutes, takes one screenshot and asks the model
whether the target state was reached. If so, fire the watch
through the shared completion path; if not, wait and
re-check. The tab-status subscription stays the primary
trigger throughout, so if the status transition does arrive
it resolves the watch and tears the backstop down via
removeWatcherAuxiliaries.

Smarter than a continuous poll: - The quiet clock is reset
by textViewDidFindDirtyRects, so an
  actively-rendering session never bills a screenshot.
  PTYSession now exposes timeSinceScreenContentsLastChanged
  and screenContentsLastChangedAt for this.
- After a not-reached verdict, the screen-change stamp is
remembered and
  the check is not repeated until the screen actually
  changes, since an identical screen yields an identical
  verdict.

armScreenEscalation is a no-op for .screenPoll watchers
(they run their own poller; arming would orphan it) and for
.working targets (a quiet screen is evidence of idle, not
work, so a check could never confirm it).

Prompt caching previously marked only the system block and
the last tool definition, so the conversation history (the
bulk of an orchestration request, dominated by accumulated
tool results) was re-sent uncached on every turn and billed
at full input rate.

block of the last message. Anthropic reads the longest
matching prefix and excludes cache_control from the cache

system/tools breakpoints and the message breakpoint share
it.

The live prompt-caching test asserted call 2 re-created zero
cache; that held only when system+tools were the sole cached
segment. With the rolling marker, call 2 legitimately writes
a small entry for its changed last message while reading the
shared prefix, so the assertion now requires the read to
dominate the tail write.

Regenerate the Anthropic live-test cassettes: every
Anthropic request body changed, so the prior recordings no
longer match and are removed in favor of freshly recorded

2026-06-19: Preserve folds when a non-alt-screen program
repaints from the top

Programs like Claude Code that do not use the alternate
screen buffer repaint on launch by homing the cursor and
drawing over whatever was on screen. The existing
saveScreenToScrollbackIfCursorMovedAboveCommandOutput
preserves content in that case, but only with shell
integration, and it bails when the program drew before
homing (appendedTextSinceCommandExecuted is set), which is
exactly what Claude does. When a fold sat in the overwritten
region its cells were destroyed while its mark survived,
leaving an orphaned unfold icon in the margin.

Add saveScreenToScrollbackIfCursorMovedAboveFold, anchored
on folds rather than the running command's output, so it
works without shell integration and regardless of whether
the program has already drawn. When the cursor moves at or
above the bottommost in-grid fold, scroll the fold and the
content above it into history, preserving the fold mark at
its absolute position.

The check runs on every cursor-positioning command, so it
reads a cached bottommost-fold absolute line
(_bottommostFoldAbsLine) instead of querying the interval
tree. Absolute coordinates are stable when a line moves
between the line buffer and the grid, so ordinary scrolling
needs no invalidation. The cache is invalidated only by

2026-06-18: Make "Bypass Terminal" reach Cocoa for
function/arrow/keypad keys. Issue 12884.

A key bound to "Bypass Terminal" is supposed to skip the
terminal and let macOS handle the keystroke normally. This
worked for ordinary keys but not for function, arrow, or

2026-06-15: Fix live filter dropping content delivered
mid-search

When a line was delivered to an AsyncFilter while a search
pass was already in progress (lastPosition == nil,
context.status == .Searching),

2026-06-09: Fix dropped completion handler in
iTermAsyncStringConversion

The conversion is dispatched to a background queue from
init, but callers set completionHandler after init returns
(both VT100Parser and the unit test do). If the background
work finished before the handler was assigned, the handler
was never called: VT100Parser's sOutstandingPreconvertBytes
backpressure counter would never be decremented for that
conversion, and testAsyncCompletionHandlerCalled would time
out. The race was timing-dependent and only showed up on the
slower CI scheduler.

it if the work already finished. Also makes
completionHandler access thread-safe (previously read on the
background queue while written on the caller thread).

2026-06-05: Git diff menu: exclude untracked files from
per-file statuses

The workgroup code-review/diff feature requests git state
with includeDiffStats=YES, which enabled rename detection in
the service-side status walk. Combined with
INCLUDE_UNTRACKED, libgit2 hashed the contents of every
untracked file in the tree for rename similarity. In a
working copy littered with large untracked files this
exceeded the git timeout, so the poll never returned: the
diff toolbar stayed empty and the diff peer hung on "Diff
session is waiting for changes".

dirty/adds/deletes, and a separate fileStatuses pass
(untracked excluded, rename detection over tracked files
only) for the diff menu. The diff menu already filtered
untracked entries out of its display, so dropping them at
the source changes no visible behavior there while
eliminating the expensive hashing.

Because untracked files no longer reach the menu, they no
longer constrain the shared-directory prefix used to shorten
displayed paths; labels for tracked changes now shorten
against only the tracked files, which is more predictable
than depending on incidental untracked clutter. Updated the
now-stale comment in CCDiffSelectorItem and extracted the
prefix/shortening logic into testable helpers.

The status bar component is unaffected: it uses
includeDiffStats=NO (only the workgroup sets it YES) and
reads dirty/branch/ahead, not fileStatuses, so its code path
and counts are unchanged.

Adds integration tests that build real git repos in temp
dirs and read them back through iTermGitClient, plus unit
tests pinning the diff menu's path-shortening behavior.

observation

register_watch previously relied entirely on tab-status (OSC

machine-readable status (plain TUIs, coding agents like

  unregister/terminate/teardown and restarted after

2026-06-01: Add disk wire logger for raw AI API traffic

New advanced setting aiChatRawWireLogging captures every
byte sent to and received from the AI vendor: full request
headers, request body, streamed SSE chunks as they arrive,
the final response or error, and elapsed time. Writes to
~/Library/Application
Support/iTerm2/AIChatWire/aichat-wire-<stamp>- pid<N>.log,
one file per app launch.

Hooked in at iTermAIClient.request, below the JS plugin
boundary, so the bytes logged are the bytes the plugin is
about to ship (or just received), not a post-parse view that
might mask the bug being debugged. A per-call UUID stamps
every record so concurrent calls are untangled in the log.
The existing test-harness liveObserver and the new logger
coexist; both observe the same wrapped stream.

Records are written on a serial queue with one shared file
handle opened lazily on first use; off-path cost is one
isEnabled check.

WARNING (documented in the setting description): the log
captures Authorization / X-Api-Key headers and prompt +
response content verbatim. The whole point is full fidelity
for debugging, so no redaction at the write layer. The
companion parser script redacts auth headers only at display
time.

Companion: tools/parse_ai_wire_log.py reconstructs streamed
Anthropic responses back into the content blocks the
non-streaming API would have returned in one shot
(text_delta fragments glued together, input_json_delta
fragments concatenated and JSON-decoded per tool_use block),
and renders the request side (model, system, tools, message
thread with tool_use / tool_result blocks broken out).
Supports --call <UUID> filtering, --raw passthrough, and
--max-body truncation.

2026-06-01: Treat any two localhosts as the same host on
host change

The host-change heuristics in screenCurrentHostDidChange
compared hostnames as bare strings, so a local .local name
drift (e.g. mDNS renaming MacBook-Pro-2.local to
MacBook-Pro-3.local) or a sudo/su could be seen as a real
host change. That spuriously offered to reset terminal state
and to restore the tab and window title.

Short-circuit the heuristics when both the previous and new
hosts have a frozen locality of localhost, regardless of
name or username.

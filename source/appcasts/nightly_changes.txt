2026-05-05: Propagate DB errors from DatabaseBackedArray
mutations

removeAll(where:) used try? on the per-element remove, which
left elements unchanged on failure but still recorded the
index as removed and never advanced past it  risking an
infinite loop and a corrupt IndexSet on persistent DB
errors. set(at:) had the same try? anti-pattern, leaving
in-memory state ahead of the DB. Both now propagate errors;
ChatListModel swallows and logs at the one UI call site.

2026-05-05: Fix UAF risk in Core Text run enumeration
fallback paths

When CTRunGetGlyphsPtr or CTRunGetStringIndicesPtr return
NULL, we allocate an NSMutableData and hand its bytes to the
closure. The NSMutableData was a strong local with no
further uses after the
.mutableBytes read, so ARC was free to release it before the closure
ran, leaving a dangling pointer. Mark the backing buffers
NS_VALID_UNTIL_END_OF_SCOPE so they survive the closure
call.

2026-05-05: Fix retain-cycle leaks in DLL-backed caches and
LRU bump bug

The doubly-linked list used by iTermCache and Swift
LRUDictionary had both prev and next as strong references,
leaking every cached entry once the owning cache was dropped
(the parallel dictionary's release wasn't enough  adjacent
nodes retained each other). Make the back-pointer weak in
both implementations; the dict and forward chain still keep
every node reachable.

Also fix LRUEvictionPolicy.bump, which removed and
re-appended a node but left nodesByKey pointing at the
detached old node. A later re-insert or delete then
corrupted list head/tail and changed which element was
treated as LRU.

Adds regression tests for both leaks and the bump bug.

2026-05-05: Fix dangling-target crash in Undo after closing
a split pane

NSTextView with allowsUndo=YES registers
_undoRedoTextOperation: actions on the responder chain's
undo manager, which for these views was the window's.
NSUndoManager retains its action targets unowned(unsafe), so
when a split pane was closed the note, session note, or chat
input text views inside it were deallocated while their
action registrations stayed in the window's undo stack as
dangling pointers. The next EditUndo (or Z) sent
_undoRedoTextOperation: to freed memory and crashed in
objc_msgSend.

Same bug also hijacked the menu title: the window's undo
manager reported canUndo=YES because of the stale
registration, so iTermApplicationDelegate's
responderForMenuItem returned the window and the item stayed
labeled "Undo" / "Undo Typing" instead of falling through to
"Undo Close Session".

Each of the three text views now provides its own
NSUndoManager (via undoManagerForTextView: on
PTYNoteViewController, and an override of undoManager on
SessionNoteTextView and ChatInputTextView). The undo stack's
lifetime is now bound to the text view's, so a closed pane
can't leave registrations behind.

2026-05-05: Fix graph db recovery rowid leak

The recovery path in trySaveEncoder aliased the failed
save's tree into a fresh recovery encoder and then called
eraseRowIDs in place, mutating record instances that other
code (notably the previous-revision tree, via the
unchanged-generation alias branch in iTermGraphDeltaEncoder)
still held. Combined with iTermOrderedDictionary
byMapping:'s "first occurrence wins" deduplication in
reallySave's enumerate, that left some records with
rowid=nil permanently in the published self.record. The next
save then tripped MissingRowID at iTermGraphDatabase.m:376.

Recovery now deep-copies originalEncoder.record into a tree
it owns exclusively (no shared instances), with rowid=nil
throughout and siblings deduplicated by (key, identifier) so
reallySave's INSERT pass assigns a fresh rowid to every
record.

2026-05-05: Fix multi-second hang when closing Settings
window

Each PreferenceInfo registered for
kPreferencePanelDidLoadNotification in -init and removed
itself in -dealloc. Closing the panel cleared the keyMap,
deallocating thousands of PreferenceInfos and producing

2026-05-04: Refresh stale session hostname variable on first
OSC RemoteHost

didFinishInitialization seeds
iTermVariableKeySessionHostname from gethostname(), which
can be a transient DHCP name (AT&T's gateway hands out bare
"Mac" via option 12). The OSC RemoteHost handler only fired
screenCurrentHostDidChange when the new mark differed from
the interval tree's last-line host, so when the shell's
later OSC matched what restoration had already parked in the

init-time value forever.

that. First OSC of a session always pushes once;
steady-state repeats remain no-ops, so no extra pause
mid-prompt.

2026-05-04: Move OpenDirectory user-shell lookup off the
main thread

Diagnosed from a sample where 100% of main-thread time
(3865/3865 samples) was blocked in +[iTermOpenDirectory
userShell] opendirectoryd via a Run Command trigger firing
on every cadence tick. The OD query is now hosted in the
pidinfo XPC service (wrapped in performRiskyBlock so a
wedged daemon is detected and recovered from instead of
burning a thread in the main app), and the main-app entry
point caches the result behind an iTermPromise.

Layout: - iTermOpenDirectory.{h,m} declares only
+performBlockingLookup,
  built into both the main app and pidinfo. Pidinfo links
  OpenDirectory.framework and calls it from inside its
  watchdog.
- iTermOpenDirectory+MainApp.{h,m} adds the (MainApp)
category
  with +userShell and +prime, built into the main app only.
  Each call kicks a fresh background lookup via the slow-op
  gateway and waits up to 20ms for it; if it doesn't return
  in time the cached value from the previous successful
  query is returned.  First call (no cache yet) blocks until
  the lookup settles.
- iTermSlowOperationGateway gains
fetchUserShellWithCompletion:,
  which always invokes its completion exactly once on a
  non-main queue (avoiding deadlock when a main-thread
  caller is blocked in promise wait) with a 15s
  defense-in-depth timeout for the case where an XPC
  connection drops mid-call.
- +prime is called from applicationWillFinishLaunching: with
a
  self-rescheduling catchError so the cache is populated
  before any synchronous caller needs to block.

The closure-vs-assignment race in the previous draft is
gone: gCurrentLookup is never written from the gateway
completion; inFlightLookupLocked rotates the promise
synchronously when the old one has settled (hasValue covers
both fulfill and reject).

2026-04-29: Fix quadratic main-thread hang on tab style
preference changes

PSMTabBarControl._removeCellTrackingRects iterated every
cell on every layout pass and called [NSNotificationCenter
removeObserver:cell] on each one. Cells were never
registered as notification observers (the line is vestigial

total observer count).

when the tab style hadn't actually changed.

2026-04-29: Fix RTL Bidi tests by pumping runloop until
cache catches up

The bidi preference is read via [iTermPreferences
bidiEnabled], a fast-path cache invalidated by an
async-to-main-queue KVO observer.  Setting the pref
synchronously in a test and then reading the cache on the
same call stack saw the stale value, because the dispatched
cache update hadn't run yet.

This is technically a race in production too, but the only
realistic caller pattern that hits it is exactly what the
tests do. Settings UI writes are followed by separate
runloop ticks before any read.

The test fix pumps the main runloop after each setBool until
the cached value catches up, with a 0.5s timeout for safety.

iTermTLVEncoder directly via the bridging header import,
replacing the duplicated manual TLV blob builder I had
introduced as a workaround.

2026-04-25: Dispatch user-defaults observers async to main
to avoid re-entrancy

CFPreferences delivers buffered KVO notifications inline on
whatever thread reads a default. Synchronous observer blocks
could (a) run on that arbitrary thread and (b) re-enter
arbitrary code during a pref read. The latter caused a crash
where a screen mutation side effect read a pref, the
observer chain reached -[NSApp setAppearance:], and the
appearance change tried to join screen threads while the
side effect was still running.

iTermUserDefaultsObserver and NSUserDefaults+iTerm now
dispatch registered blocks async to the main queue. Both
also synchronize their shared block storage since KVO can
fire off-main while a registration runs on main.

iTermKeyMappings.setGlobalKeyMap: had a post-write assert
that depended on the observer running synchronously to
update the gGlobalKeyMapping cache; it now updates the cache
directly. removeAllGlobalKeyMappings goes through
setGlobalKeyMap: for consistency.

switched to performBlockWithJoinedThreadsReentrantSafe: to
defend against AppKit spontaneously firing
viewDidChangeEffectiveAppearance during a side effect (see

2026-04-23: Fix git status bar component timing out and
silently failing

Three independent issues caused the git status bar to show
empty in large repos like chromium:

1. Ahead/behind walk was unbounded.
getCountsFromRef:pull:push: used a
   hand-rolled git_revwalk that pushed only the tip and
   broke when it happened to encounter the merge base in
   topological order. With no git_revwalk_hide call, it
   could traverse hundreds of thousands of unrelated
   ancestors first. Replaced with git_graph_ahead_behind,
   the libgit2 API sized for this exact job. Also freed an
   upstream_ref leak.

2. pidinfo's wedge watchdog was hardcoded to 10s (release) /
30s (debug)
   for every call, including git state requests. Once the
   status scan started taking longer than that, the wedge
   path delivered (nil, YES) to the client before the
   subprocess could finish, and the valid reply that arrived
   later was dropped. Parameterized the watchdog: git
   requests now pass gitTimeout + 5s so the subprocess can
   actually run to completion under its own SIGKILL
   watchdog.

3. The UI showed nothing when polls genuinely timed out.
Threaded a
   timedOut flag through the XPC reply, the gateway, the
   poll worker, and iTermGitPoller (separate from
   state-is-nil, which also covers "no git repo here").
   iTermStatusBarGitComponent now shows " timed out" when
   we've never successfully fetched for the current
   directory and the last poll timed out. Clicking offers to
   double the timeout (via new DEFINE_SETTABLE_FLOAT on
   gitTimeout) and immediately retries. Per-directory flags
   reset on cd so a fast-then-slow sequence surfaces the
   error correctly, and in-flight results for stale paths
   are discarded.

2026-04-11: Fix use-after-free crash in _NSFullScreenSpace

and DECSWL (ESC #5) for double-width and double-height line

Data model ---------- Add iTermLineAttribute enum
(SingleWidth, DoubleWidth, DoubleHeightTop,
DoubleHeightBottom) to iTermMetadata and to
iTermExternalAttribute.h.  In the live grid, lineAttribute
is a field of iTermMetadata on each line. When flushing to

attribute from the uniform per-character values.

cell after each real character on double-width lines,
analogous to DWC_RIGHT for CJK. Spacers are stripped when
entering LineBuffer and re-inserted on read-back.

We do not attempt to reflow double-width characters at the
end of a line. That's because double-width compounded by
full-width (e.g., Emoji) means a character can occupy 4
cells and it would complicate line buffer location <->
coordinate conversions too much.

VT100OtherParser as DECDHL, DECSWL, and DECDWL tokens. Route
through VT100Terminal to a new terminalSetLineAttribute:
delegate method.

current cursor line. VT100Grid clamps cursor positions to
even cells on double-width lines, scales cursor movement by
2x, and fills odd cells with DWL_SPACER during erase
operations.

iTermTextDrawingHelper applies CGContextScaleCTM(ctx, 2.0,
1.0) for horizontal doubling, sets the text matrix vertical
scale to 2x for double-height lines, clips to one row to

halves. Emoji on double-height lines use CTM scaling since
color bitmap fonts ignore the text matrix.

Metal rendering --------------- iTermCharacterSource gains
drawHScale/drawVScale. The glyph subclass applies 2x
horizontal CTM and 2x vertical text matrix for

deduplicateDECDHL flag to skip redundant bottom halves

serializes all data inline. The main interval tree already

missed. The restore side falls back to the legacy format for
backward compatibility.
